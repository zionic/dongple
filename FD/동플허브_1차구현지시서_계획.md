# 동플허브_1차구현지시서_계획

## 1. 목적
- 이 문서는 동플 허브의 1차 구현 범위를 고정하고, 구현 담당자가 바로 개발에 들어갈 수 있도록 작업 순서를 정리한 실행 지시서다.
- 이번 1차 목표는 “실시간 상태 공유 + 지도 화면 + TourAPI 이벤트 기본 노출 + 메인 진입 구조 정리”까지다.
- 커뮤니티 고도화, 신고, 통계 배치, 관리자 기능은 1차 범위에서 제외한다.

## 2. 1차 구현 목표

### 필수 구현
- 실시간 상태 목록 조회
- 실시간 상태 등록
- 상태 “맞아요” 인증 증가
- 지도 화면에서 상태 목록과 마커 동기화
- TourAPI 기반 공식 이벤트 목록/상세 조회
- 메인페이지에서 상태/이벤트 요약 노출

### 제외 항목
- 본인인증 연동
- `profiles`, `reports`, `stats_daily`, `event_links` 구현
- 관리자 콘솔
- 실시간 WebSocket/SSE
- 머신러닝 기반 분류기
- 신고/제재 자동화

## 3. 구현 순서

### 1단계. DB 및 기본 데이터 접근 계층
- `schema.sql` 기준으로 Supabase 테이블을 생성한다.
- `live_status`, `status_verifications`, `events`, `events_ext`, `raw_items`, `posts`를 기준 테이블로 사용한다.
- `events_ext.event_id`는 `BIGINT` 기준으로 유지한다.
- `increment_verified_count` RPC를 사용할 수 있게 배포한다.
- `seed_suwon_events.sql`로 테스트용 이벤트 데이터를 주입한다.

### 2단계. 상태 공유 기능
- `statusService.ts`를 만든다.
- 필요한 기능:
  - 활성 상태 목록 조회
  - 상태 등록
  - 상태 인증 수 증가
  - 지도용 좌표 포함 목록 조회
- 기본 정렬은 `created_at DESC` 또는 활성 상태 우선 기준을 사용한다.
- `expires_at > now()` 조건의 데이터만 우선 노출한다.

### 3단계. 상태 보드 UI
- `features/status/components/LiveStatusBoard.tsx`를 만든다.
- 필요한 기능:
  - 상태 카드 목록
  - `맞아요` 버튼
  - `아니에요` 또는 다른 상황 제보 진입
  - 상세 메시지 보기
- 상태 카드는 `여유`, `보통`, `혼잡`, `요청` 4개 상태에 따라 시각적으로 구분한다.

### 4단계. 지도 화면 UI
- `src/app/map/page.tsx`를 만든다.
- 필요한 기능:
  - 지도 렌더링
  - 바텀시트 리스트
  - 리스트-마커 상호작용
  - 상태 제보 진입
- 지도 라이브러리는 구현팀이 실제 저장소 환경에 맞게 선택하되, 페이지 내부에서 직접 데이터 로직을 가지지 말고 `statusService.ts`를 호출한다.

### 5단계. TourAPI 이벤트 기능
- `eventService.ts`를 만든다.
- 필요한 기능:
  - 이벤트 목록 조회
  - 이벤트 상세 조회
  - `events_ext.meta` 포함 데이터 정리
- 1차 구현에서는 실제 TourAPI 실시간 동기화보다 `schema.sql + seed_suwon_events.sql` 기반 동작을 우선한다.
- 추후 배치 수집 로직은 2차 구현으로 넘긴다.

### 6단계. 메인페이지 구성
- `src/app/page.tsx`를 만든다.
- 필요한 구성:
  - 서비스 소개/CTA
  - 실시간 상태 요약 섹션
  - 공식 이벤트 요약 섹션
  - 지도 화면/상태 제보로 이동하는 진입 버튼
- 메인페이지는 1차에서 “진입점 역할”만 안정적으로 수행하면 된다.

## 4. 권장 구현 구조

### 권장 폴더 구조
```text
src/
  app/
    page.tsx
    map/page.tsx
    events/page.tsx
  features/
    status/
      components/
        LiveStatusBoard.tsx
        StatusCard.tsx
        StatusReportModal.tsx
    events/
      components/
        EventCard.tsx
        EventList.tsx
  services/
    statusService.ts
    eventService.ts
    supabase/
  shared/
    types/
    ui/
```

### 서비스 분리 원칙
- 페이지는 데이터 로직을 직접 가지지 않는다.
- `services/*`는 Supabase 호출과 응답 정리를 담당한다.
- `features/*`는 UI와 사용자 상호작용을 담당한다.

## 5. 기능별 완료 기준

### 상태 공유 완료 기준
- 상태를 등록하면 목록에 즉시 보인다.
- `맞아요` 클릭 시 `verified_count`가 증가한다.
- 만료된 상태는 기본 목록에서 제외된다.

### 지도 화면 완료 기준
- 상태 목록과 지도 마커가 같은 데이터를 바라본다.
- 마커 클릭 시 리스트가 해당 항목과 연결된다.
- 리스트 항목 클릭 시 지도 위치가 해당 장소로 이동한다.

### 이벤트 기능 완료 기준
- 시드 데이터 기준으로 이벤트 목록이 정상 노출된다.
- 이벤트 카드에 제목, 날짜, 썸네일, 위치가 보인다.
- 이벤트 상세 진입 시 `events_ext.meta`를 확장 가능 구조로 읽을 수 있다.

### 메인페이지 완료 기준
- 상태 요약과 이벤트 요약이 각각 1개 이상 노출된다.
- 사용자가 상태 보드 또는 지도 화면으로 자연스럽게 이동할 수 있다.

## 6. 구현팀 주의사항
- 현재 문서 폴더에는 실제 `src/...` 앱 소스가 없으므로, 구현팀은 별도 앱 저장소에서 위 경로를 생성해야 한다.
- `schema.sql`은 MVP 기준이다. 1차 구현 중 마스터 스펙의 장기 테이블까지 한 번에 넣지 않는다.
- 인증/익명화의 완전한 운영 버전은 2차 구현으로 넘기고, 1차에서는 구조만 해치지 않는 선에서 단순화한다.
- TourAPI 실시간 배치 연동은 2차로 넘기고, 1차에서는 이벤트 조회 UI와 DB 구조를 먼저 안정화한다.
