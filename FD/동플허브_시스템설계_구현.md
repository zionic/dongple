# 동플허브_시스템설계_구현

## 1. 문서 목적
- 이 문서는 동플 허브를 실제 구현 가능한 단위로 분해한 개발자 전달용 시스템 설계서다.
- 마스터 스펙의 장기 목표를 유지하되, 현재 저장소 기준에서는 문서/DB 초안 중심 구조에 맞춰 MVP부터 확장 가능한 형태로 정리한다.
- 구현 담당자는 이 문서를 기준으로 프론트, 백엔드, 데이터 계층의 책임을 분리해 작업한다.

## 2. 제품 구조 요약
- 동플 허브는 지역 기반 실시간 상황 공유 서비스다.
- 1차 구현 축은 `실시간 상태 공유`, `커뮤니티 게시물`, `공식 이벤트 연동` 3개로 나눈다.
- 본인인증, 익명화, 신고, 통계 배치 같은 항목은 구조를 고려하되 단계적으로 확장한다.

## 3. 시스템 모듈

### 3.1 상태 공유 모듈
- 사용자들이 특정 장소의 현재 혼잡도/상태를 제보한다.
- 상태값은 `여유`, `보통`, `혼잡`, `요청` 중심으로 운영한다.
- 주요 데이터는 `live_status`, `status_verifications`를 사용한다.
- UI 진입점은 상태 보드와 지도 화면이다.

### 3.2 커뮤니티 모듈
- 익명 또는 닉네임 기반으로 게시글을 작성하고 읽는다.
- 주요 데이터는 `posts`를 사용한다.
- 향후 `reports`, `comments`, `likes`로 확장 가능하도록 설계한다.

### 3.3 공식 이벤트 모듈
- TourAPI를 통해 공식 축제/행사 데이터를 수집한다.
- 주요 데이터는 `events`, `events_ext`, `raw_items`를 사용한다.
- 사용자 제보와 공식 이벤트를 연결해 신뢰도 보정에 활용한다.

### 3.4 인증/프로필 모듈
- 현재 단계에서는 Supabase Auth 또는 외부 인증을 전제로 인터페이스만 정의한다.
- 장기적으로는 `profiles` 중심으로 닉네임, 지역 인증, GPS 인증 상태를 보관한다.
- 마스터 스펙상의 `users`는 플랫폼 인증 계층 개념으로 유지하고, 실제 앱 구현은 `auth.users + profiles` 구조를 우선 검토한다.

### 3.5 운영/신고/집계 모듈
- 현재 문서에는 장기 구조만 정의한다.
- 향후 `reports`, `stats_daily`를 추가해 악성 제보 대응 및 지역별 집계를 운영한다.

## 4. 권장 구현 아키텍처

### 4.1 저장소 기준
- 현재 폴더는 문서/DB 초안 중심 저장소다.
- 실제 앱 소스는 별도 저장소이거나 후속 생성 대상으로 본다.
- 따라서 본 문서의 파일/폴더 구조는 구현팀이 앱 저장소에서 생성할 기준 구조다.

### 4.2 계층 구조
- `UI Layer`: 페이지, 컴포넌트, 입력 폼, 지도/리스트 상호작용
- `Feature Layer`: 상태 공유, 이벤트, 게시글 도메인별 프론트 로직
- `Service Layer`: Supabase, TourAPI, 캐시, 인증 인터페이스
- `Data Layer`: PostgreSQL/Supabase 테이블, RLS 정책, RPC 함수

### 4.3 권장 앱 폴더 구조

## 5. 구현 우선순위

### 1단계. 상태 공유 MVP
- `live_status` 목록/등록/상세/인증 수 증가를 구현한다.
- 상태 보드와 지도 화면을 먼저 연결한다.
- `increment_verified_count` RPC를 활용해 “맞아요” 액션을 구현한다.

### 2단계. 공식 이벤트 연동
- TourAPI 수집 로직과 `events`, `events_ext`, `raw_items` 연결을 구현한다.
- 이벤트 리스트/지도 표시와 상태 공유의 연결 지점을 만든다.

### 3단계. 게시글/익명화
- `posts`를 기반으로 게시글 읽기/쓰기/익명 노출 규칙을 구현한다.
- 향후 `public_id`, 프로필, 신고 모델 추가를 고려해 인터페이스를 분리한다.

### 4단계. 메인 통합/V2 전환
- 메인페이지 보완안을 반영한다.
- V2 메인페이지를 기본 루트로 승격한다.

### 5단계. 운영 기능 확장
- `profiles`, `reports`, `stats_daily`를 추가한다.
- 관리자용 제재, 집계, 신뢰도 검토 흐름을 확장한다.

## 6. 핵심 설계 원칙
- 마스터 스펙은 장기 목표 문서로 유지하고, 구현 범위는 기능 설계 문서와 SQL에서 점진적으로 맞춘다.
- DB 설계는 MVP를 먼저 안정화한 뒤 확장 테이블을 추가한다.
- 프론트 구조는 페이지 기준보다 도메인 기준으로 나눈다.
- 외부 연동 데이터는 `raw_items`에 보존하고, 서비스에 노출할 정제 데이터는 `events/events_ext`로 분리한다.
- 현재 폴더에 없는 `src/...` 경로는 구현팀이 실제 앱 저장소에서 생성한다.

## 7. 구현팀 전달 메모
