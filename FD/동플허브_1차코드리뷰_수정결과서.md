## 4. 우선순위별 수정 항목

### 4.1 P1-1 상태 인증 수 증가 로직 무결성 보완
- 대상 파일: `C:\dongple_workspace\dongple\src\services\statusService.ts`
- 관련 구간: `verifyStatus`

#### 현재 문제
- `status_verifications` insert가 실패해도 `increment_verified_count` RPC가 계속 호출된다.
- 현재 스키마에는 `(status_id, user_id)` 중복 방지 제약이 없어 동일 사용자가 반복 클릭하면 인증 수를 계속 올릴 수 있다.
- insert 실패와 카운트 증가가 분리되어 있어 데이터 무결성이 깨질 수 있다.

#### 수정 목적
- 사용자 1명당 같은 상태에 대해 1회만 인증되도록 보장한다.
- 검증 기록 저장과 카운트 증가를 원자적으로 처리한다.

#### 권장 수정 방향
- DB 레벨에서 `status_verifications(status_id, user_id)` 유니크 제약을 추가한다.
- 가능하면 `verifyStatus`를 단일 RPC로 변경한다.
  - 예시: `verify_status_once(p_status_id uuid, p_user_id text)`
  - 내부에서 insert 성공 시에만 `live_status.verified_count` 증가
  - 중복 인증이면 증가 없이 종료
- 프론트 `verifyStatus`는 해당 RPC 호출만 수행하도록 단순화한다.
- 임시 대응만 할 경우에도 insert 실패 시 RPC를 호출하지 않도록 순서를 고친다.

#### 완료 기준
- 동일 `status_id`, `user_id` 조합으로 여러 번 호출해도 `verified_count`는 1회만 증가한다.
- insert 실패 시 `verified_count`가 증가하지 않는다.
- 중복 인증 시 사용자 메시지가 정상적으로 안내된다.

### 4.2 P1-2 지도 마커 및 오버레이 정리 로직 추가
- 대상 파일: `C:\dongple_workspace\dongple\src\app\map\page.tsx`
- 관련 구간: `renderMarkers`

#### 현재 문제
- `markers`, `expandedCardId`, `sheetHeight` 변경 때마다 마커를 다시 렌더링한다.
- 이전 `naver.maps.Marker`를 `setMap(null)`로 제거하지 않는다.
- `createRoot`로 만든 공식 이벤트 오버레이 React 루트를 unmount하지 않는다.
- 결과적으로 마커 중복, 클릭 영역 중첩, 메모리 누수, 렌더링 비용 증가가 발생할 수 있다.

#### 수정 목적
- 지도 상 마커와 오버레이를 상태 변화에 맞게 정확히 재구성한다.
- 대량 마커 환경에서도 성능 저하를 줄인다.

#### 권장 수정 방향
- `markersRef`를 단순 배열이 아니라 정리 가능한 객체 배열로 관리한다.
  - 예시: `{ marker, root, element }`
- `renderMarkers` 시작 전에 기존 마커를 순회하며 정리한다.
  - `marker.setMap(null)`
  - `root?.unmount()`
- 공식 이벤트는 `fetchOfficialEvents()`를 렌더 함수 내부에서 반복 호출하지 말고 별도 상태로 캐싱한다.
- `sheetHeight` 변화만으로 마커 전체를 재생성하지 않도록 의존성을 분리한다.
  - 선택 상태만 바뀌면 기존 마커 스타일만 업데이트하거나 선택 마커만 갱신

#### 완료 기준
- 바텀시트 높이 변경, 카드 선택/해제, 데이터 재조회 시 지도 마커 개수가 비정상적으로 증가하지 않는다.
- React DevTools 및 브라우저 메모리 관찰 시 오버레이 루트가 누적되지 않는다.
- 동일 이벤트/상태에 대한 마커가 중복 표시되지 않는다.

### 4.3 P1-3 공식 이벤트 목록 조회에 확장 필드 조인 반영
- 대상 파일: `C:\dongple_workspace\dongple\src\services\eventService.ts`
- 관련 구간: `fetchOfficialEvents`

#### 현재 문제
- `fetchOfficialEvents`는 `events` 테이블만 조회한다.
- `meta`, `source`는 `fetchEventDetail`에서만 채워지는데, 실제 화면 대부분은 `fetchOfficialEvents`를 사용한다.
- 메인 섹션, 지도, 이벤트 페이지에서 확장 메타데이터가 누락된다.

#### 수정 목적
- 공식 이벤트 목록 경로에서도 확장 메타데이터를 함께 전달한다.
- 요구사항에 명시된 `events + events_ext` 결합을 실제 UI 경로에 반영한다.

#### 권장 수정 방향
- `fetchOfficialEvents`에서 `events_ext`를 함께 조회하도록 변경한다.
- 구현 방식 예시
  - Supabase 관계 조회 사용 가능 시 select에 관계 포함
  - 관계 조회가 어려우면 `events`, `events_ext`를 각각 조회 후 `event_id` 기준으로 병합
- 반환 타입에서 `meta`, `source`를 항상 일관되게 채운다.
  - 기본값: `meta = {}`, `source = 'TOURAPI'`
- 지도/목록/메인 섹션에서 추가 필드가 필요한 경우 별도 재조회 없이 바로 사용할 수 있게 한다.

#### 완료 기준
- `fetchOfficialEvents()` 반환값에 `meta`, `source`가 포함된다.
- 메인 섹션, 지도, 이벤트 페이지에서 동일한 확장 필드 접근이 가능하다.
- `fetchEventDetail`와 목록 응답 간 데이터 형태가 크게 어긋나지 않는다.

### 4.4 P2-1 상태 선택 버튼의 보통 색상 토큰 정합성 수정
- 대상 파일: `C:\dongple_workspace\dongple\src\components\forms\LiveStatusCreateForm.tsx`
- 관련 구간: 상태 선택 버튼 className

#### 현재 문제
- `보통` 상태 버튼 본문 색상은 파란 계열로 변경되었지만, 선택 강조 링은 `ring-yellow-300`으로 남아 있다.
- 보드, 지도, 폼 사이에서 `보통 = Blue` 규칙이 완전히 일치하지 않는다.

#### 수정 목적
- 상태 색상 규칙을 디자인 토큰 수준에서 일관되게 맞춘다.

#### 권장 수정 방향
- `보통` 상태의 선택 링을 `ring-blue-300` 또는 이에 준하는 파란 계열로 교체한다.
- 가능하면 상태별 색상을 상수 또는 공통 토큰으로 분리한다.
  - 예시: `STATUS_THEME.normal.ring`
- `LiveStatusBoard`, `LiveStatusCreateForm`, `map/page.tsx`에서 동일 토큰을 참조하도록 구조 개선을 검토한다.

#### 완료 기준
- `보통` 상태의 배지, 텍스트, 선택 링이 모두 파란 계열이다.
- 화면별로 `보통` 상태가 다른 색으로 보이지 않는다.

### 4.5 P2-2 이벤트 카드 썸네일 실사용 경로 추가
- 대상 파일: `C:\dongple_workspace\dongple\src\app\events\page.tsx`
- 관련 구간: 이벤트 카드 이미지 영역

#### 현재 문제
- 타입과 스키마에는 `thumbnail_url`이 있으나 실제 카드에서는 항상 Ghost 아이콘만 렌더링한다.
- 이미지가 있는 데이터도 누락 데이터처럼 보인다.

#### 수정 목적
- 이미지가 있으면 실제 썸네일을 보여주고, 없거나 실패한 경우에만 fallback UI를 표시한다.

#### 권장 수정 방향
- `thumbnail_url` 존재 시 실제 이미지 렌더링을 시도한다.
- 이미지 로드 실패 시에만 Ghost fallback으로 전환한다.
- 가능하면 `next/image` 사용을 검토한다.
- 이미지가 없는 경우를 대비한 fallback UI는 유지하되, “정상 이미지 경로”를 먼저 만든다.

#### 완료 기준
- `thumbnail_url`이 있는 이벤트는 카드에서 이미지가 노출된다.
- 이미지가 없거나 깨진 경우에만 Ghost 아이콘이 표시된다.
- 목록 스크롤 시 레이아웃 점프가 크지 않다.

## 5. 권장 작업 순서
1. `schema.sql`과 Supabase 함수부터 보완해 상태 인증 무결성을 먼저 고친다.
2. `statusService.ts`를 새 RPC 기준으로 단순화한다.
3. `eventService.ts`에서 `events_ext` 병합을 반영한다.
4. `map/page.tsx`의 마커 정리 로직과 공식 이벤트 캐싱 구조를 손본다.
5. `LiveStatusCreateForm.tsx` 색상 토큰을 정리한다.
6. `events/page.tsx` 썸네일 표시 및 fallback 경로를 보완한다.

## 6. 테스트 체크리스트
- 상태 인증 버튼을 같은 계정으로 여러 번 눌러도 카운트가 1회만 증가하는지 확인
- 인증 insert 실패 상황에서 카운트가 증가하지 않는지 확인
- 지도 바텀시트 높이 조절, 카드 선택/해제 반복 시 마커 중복 생성이 없는지 확인
- 지도에서 공식 이벤트 마커 개수가 렌더링 반복 후에도 유지되는지 확인
- 메인 페이지, 지도, 이벤트 목록에서 `meta`, `source`가 공통 형태로 전달되는지 확인
- 상태 생성 폼에서 `보통` 선택 시 파란 강조색이 적용되는지 확인
- 이벤트 카드에서 이미지가 있으면 썸네일이 보이고, 없으면 Ghost fallback이 보이는지 확인

## 7. 구현 메모

- 후속 리팩터링 시 `features/events`, `features/status`, `shared/map` 계층으로 더 분리하는 것이 좋다.
